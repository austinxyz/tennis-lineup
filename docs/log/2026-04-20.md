## 本次沉淀的经验（更新 CLAUDE.md / openspec/config.yaml）

### 1. 新增 HTTP 409 错误路径范式
此次是项目首次引入 409 Conflict：状态冲突类错误（资源非空 / 资源状态不允许此操作）应区别于 400 VALIDATION_ERROR。

- **约定**：错误码 `<RESOURCE>_<CONDITION>`（如 `TEAM_NOT_EMPTY`），大写下划线
- **异常类型**：独立 `RuntimeException` 子类（如 `TeamNotEmptyException`），**不**要继承 `IllegalArgumentException`（否则会被既有的 400 handler 提前拦截）
- **Handler**：在 `GlobalExceptionHandler` 新增 `@ExceptionHandler(XxxException.class)` → `HttpStatus.CONFLICT` + 对应 `ErrorResponse`

### 2. `ErrorResponse.details` 现已为 `Object`（而非 `String`）
- 结构化细节（如 `{"playerCount": 3, "lineupCount": 2}`）可直接挂在 `details` 上，由 Jackson 序列化为嵌套 JSON 对象
- 旧的 `String` 值（即使为 `null`）仍可正常工作 —— 向后兼容
- 新增错误码时优先用 `Map.of(...)` 结构化返回，避免把多字段塞进 `message` 字符串里再让前端解析

### 3. TDD 对"回归保护测试"的豁免
Iron Law 是"没有失败测试先于生产代码"。但如果新 spec 的要求**已经被既有代码满足**（本次 `useTeams.js` 409 错误传播即是如此），那么：
- 不存在新的生产代码 → Iron Law 的前提不成立
- 新增的测试是**回归守卫**，首次即 PASS 是预期行为（不是违反 TDD）
- 仍应把该行为写成显式测试，防止未来重构退化
- 在 test-report / dev log 中**明确标注**这是 "regression guard, no new production code"，避免误读为 TDD 违规

### 4. "Dead code 镜像"反模式
`TeamList.vue` 未被 router 引用（只有 `TeamManagerView` → `TeamListPanel` 这条路径）。tasks.md 让我"同步改 TeamList.vue"，实际上这是向 dead code 复制一份保护逻辑，既耗时间也增加未来维护负担。

**新约定**：tasks.md 要求改动文件 X 时，**先验证 X 是否被 router / 父组件引用**：
- `grep -r "import ['\"].*X" frontend/src` 或看 `router/index.js`
- 如果 X 是 dead code → 在 proposal / review 阶段明确写明"X 未挂接，提议删除而非修改"，而不是默默镜像
- 真要保留镜像（防后续启用不合规），在 tasks.md 注明原因

### 5. `openspec archive` 其实自带 spec sync
tasks.md 11.1 条目提到 "先跑 openspec-sync-specs 再 archive" —— 实际上 `openspec archive <name>` **已经会把 delta specs 合并**到 `openspec/specs/<capability>/spec.md`，无需分两步（`openspec archive --help` 显示 `--skip-specs` 选项确认这点）。

**约定更新**：
- 直接 `openspec archive <name>` 一条命令搞定，不需要分步
- 但务必**先 git commit**（含代码 + `openspec/changes/<name>/`）**再** archive，否则 archive 产生的 `openspec/specs/` 变更与实现代码混在一个 commit 里，history 难读。正确顺序：
  1. `git add -A && git commit` → 2. `git push` → 3. `openspec archive <name>` → 4. `git commit "docs: archive <name>"` → 5. `git push`

### 6. 新增 error-response 契约变动 → 需同步 `openspec/config.yaml` 的 `api` 规则
当前 `config.yaml` line 278："新异常类型继承 IllegalArgumentException（400）或直接用 NotFoundException（404）" —— 只覆盖 400/404。本次改动后应增补 409 分支。
