## Context

当前球员 CRUD 已完整实现（后端 70 tests 通过，前端 115 tests 通过）。`Player` 模型有 `utr: double`，但 UI 层未强制两位小数精度，也没有 profileUrl 字段。球员列表仅支持单条编辑（通过弹出表单）。

## Goals / Non-Goals

**Goals:**
- UTR 输入/显示统一精确到两位小数
- 球员记录可选存储 UTR 官网个人页 URL，列表页可点击直接跳转
- 队员列表支持批量内联编辑 UTR，减少多次弹窗操作

**Non-Goals:**
- 自动从 UTR 网站抓取最新 UTR（不做爬虫/API 集成）
- URL 格式验证（仅做可选字段，不强制格式）
- 批量编辑其他字段（姓名、性别、verified 等）

## Decisions

### 决策 1：批量编辑方式 — 内联编辑 vs 独立弹窗

**选择**：内联编辑（在球员列表行内切换为输入框）

**理由**：用户场景是"赛前快速更新所有人的 UTR"，内联编辑一次看到全部球员、对比修改效率最高。独立弹窗逐一打开反而更慢。

---

### 决策 2：批量保存策略 — 并发多请求 vs 新增批量 API

**选择**：前端并发调用现有 `PUT /api/teams/{id}/players/{pid}`，无需新增后端 API

**理由**：球队规模通常 8–15 人，并发 8–15 个 PUT 请求完全可接受。避免为边缘场景引入新 API 端点，减少后端改动范围。仅对实际修改过的 UTR 发起请求（跳过未改动的球员）。

---

### 决策 3：UTR 精度 — 后端四舍五入 vs 前端控制

**选择**：前端控制（`step=0.01`，显示 `toFixed(2)`），后端不做精度处理

**理由**：后端存储 Java `double`，精度由 IEEE 754 保证。业务精度需求（两位小数）在输入层控制更直观，避免隐式服务端修改用户数据。后端校验仅做范围检查（0.0–16.0），不变。

---

### 决策 4：profileUrl 字段验证 — 严格 URL 格式 vs 可选字符串

**选择**：可选字符串，不做格式校验（前端用 `<a href>` 直接跳转，非 null 时显示）

**理由**：用户自己粘贴 UTR 官网链接，URL 格式基本可信。加严格校验增加摩擦但收益低。若后续需要，可在单独 change 中加。

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|---------|
| 批量保存中部分请求失败 | 前端收集所有失败项，统一展示错误列表；成功项已保存，失败项保持高亮提示用户重试 |
| 用户在批量编辑模式误触离开页面 | 切换路由前检查是否有未保存修改，提示确认 |
| `profileUrl` 存储 XSS 风险（`<a href>`）| 前端用 `href` 属性而非 `innerHTML`，浏览器原生安全；同时设置 `target="_blank" rel="noopener noreferrer"` |

## Migration Plan

- `Player.java` 新增 `profileUrl` 字段，Jackson 反序列化时缺失字段默认 `null`，无需数据迁移
- 前端 `PlayerForm.vue` 和 `TeamDetail.vue` 修改向后兼容（profileUrl 为 null 时不显示链接）
